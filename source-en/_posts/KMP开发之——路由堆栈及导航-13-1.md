## 二 概念

### 2.1 KMP 中为什么“导航”是个难题？

1-在单平台中

|  平台   |         导航方案         |
| :-----: | :----------------------: |
| Android | NavController / Fragment |
|   iOS   |  UINavigationController  |
|   Web   |        URL Router        |
| Desktop |      Window / Scene      |

2-KMP 要求

```
一套代码
多端运行
不依赖平台原生导航体系

因此：KMP 中的导航 ≠ Android Navigation
```

### 2.2 KMP 中的导航设计目标

```
一个合格的 KMP 路由系统应满足：

1.纯 Kotlin(commonMain 可用)
2.基于 Compose
3.支持「入栈 / 出栈」
4.支持参数传递
5.支持 ViewModel 生命周期
6.可扩展多端(Android/iOS/Desktop/Web)
```

### 2.3 KMP 常见路由方案对比

|    方案     | 推荐度 |        特点        |
| :---------: | :----: | :----------------: |
|   Voyager   | 5颗星  |   官方社区最成熟   |
|  Decompose  | 4颗星  | 强架构、学习成本高 |
| 自研 Stack  | 3颗星  |    灵活但成本高    |
| Android Nav | 不推荐 |      非跨平台      |

### 2.4 Voyager 简介

1-说明

```
Voyager 是为 Compose Multiplatform 设计的导航库
```

2-核心特性

```
1.Screen = 页面
2.Navigator = 路由栈
3.支持：
 -栈式导航
 -Tab
 -BottomSheet
 -Dialog
 -ScreenModel(类似 ViewModel)
```

## 三 实战
### 3.1 基础概念速览

1-Screen(页面)

```
class HomeScreen : Screen {
    @Composable
    override fun Content() {
        Text("Home")
    }
}
```

2-Navigator(导航容器)

```
Navigator(HomeScreen())

Navigator 本质是一个 路由栈
```

3-push / pop

```
val navigator = LocalNavigator.current

navigator?.push(DetailScreen())
navigator?.pop()
```

### 3.2 KMP 项目中集成 Voyager

1-依赖(commonMain)

```
dependencies {
    implementation("cafe.adriel.voyager:voyager-navigator:1.0.0")
    implementation("cafe.adriel.voyager:voyager-screenmodel:1.0.0")
}
```

2-App 入口

```
@Composable
fun App() {
    Navigator(HomeScreen())
}
```

### 3.3 路由堆栈运行原理

```
1-Voyager 内部维护一个：
Stack<Screen>

2-操作
push() → 入栈
pop() → 出栈
replace() → 栈顶替换

3-说明：
与 Android Fragment back stack 类似，但完全跨平台
```

### 3.4 页面跳转实战

1-Home —>Detail

```
class HomeScreen : Screen {
    @Composable
    override fun Content() {
        val navigator = LocalNavigator.current

        Button(onClick = {
            navigator?.push(DetailScreen("KMP"))
        }) {
            Text("Go Detail")
        }
    }
}
```

2-Detail 页面接收参数

```
class DetailScreen(private val title: String) : Screen {

    @Composable
    override fun Content() {
        Column {
            Text("Title: $title")
        }
    }
}

说明：参数天然支持(构造函数)
```

### 3.5 返回逻辑统一处理

```
BackHandler {
    navigator?.pop()
}

说明：在 Android / Desktop / Web 行为一致
```

## 四 补充
### 4.1 路由与状态的正确关系

1-错误做法

```
路由放 ViewModel
页面自己管理堆栈
```

2-正确做法

```
Navigator 管路由
ScreenModel 管状态
```

### 4.2 路由与 ScreenModel(简述)

```
class HomeModel : ScreenModel {
    var count by mutableStateOf(0)
}

val model = rememberScreenModel { HomeModel() }

ScreenModel 生命周期 绑定 Screen
```

## 五 与 Flutter / RN 路由对比

|  框架   |      路由思想      |
| :-----: | :----------------: |
| Flutter | Navigator + Route  |
|   RN    |     JS Router      |
|   KMP   | Navigator + Screen |
| Android |      Fragment      |

说明：Voyager 的设计非常接近 Flutter

## 六 小结

```
1.KMP 路由必须跨平台
2.Voyager 是当前最优解
3.Screen = 页面
4.Navigator = 路由堆栈
5.构造函数即参数传递
6.ScreenModel 管状态，不管路由
```
