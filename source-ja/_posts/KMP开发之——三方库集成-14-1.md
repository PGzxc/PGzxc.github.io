## 二 KMP 三方库三大分类

### 2.1 纯 Kotlin Multiplatform 库(最理想)

1-特点

```
-提供 commonMain
-无平台 API
-可直接跨平台

推荐优先级最高
```

2-常见示例

|          库           |   用途   |
| :-------------------: | :------: |
|         Ktor          |   网络   |
| kotlinx.serialization |   JSON   |
|      Coroutines       |   协程   |
|         Kamel         | 图片加载 |
|        KStore         | 本地存储 |
|      SQLDelight       |  数据库  |
|        Voyager        |   路由   |
|  Coil-Compose (部分)  |   图片   |

### 2.2 半跨平台库（需 expect / actual）

1-特点

```
-common 定义接口
-平台实现能力

KMP 核心用法
```

2-示例

```
1.Toast
2.Clipboard
3.Permission
4.File
5.Device Info
6.Share
```

### 2.3 单平台库(不可直接复用)

1-特点

```
Android-only / iOS-only
只能放 platformMain
```

2-示例

```
Glide
AndroidX Camera
AVFoundation（iOS）
```

## 三 三方库集成实战

### 3.1 三方库集成的正确分层策略

1-推荐模块分层

```
shared/
 ├── commonMain
 │    ├── domain
 │    ├── ui
 │    ├── data
 │    └── thirdparty (接口)
 ├── androidMain
 │    └── thirdparty (实现)
 ├── iosMain
 │    └── thirdparty (实现)
```

### 3.2 纯 KMP 三方库集成(commonMain)—示例 1：Ktor 网络库

1-commonMain配置

```
commonMain {
    dependencies {
        implementation("io.ktor:ktor-client-core:2.3.3")
        implementation("io.ktor:ktor-client-content-negotiation:2.3.3")
        implementation("io.ktor:ktor-serialization-kotlinx-json:2.3.3")
    }
}
```

2-平台补充

```
androidMain {
    implementation("io.ktor:ktor-client-okhttp:2.3.3")
}

iosMain {
    implementation("io.ktor:ktor-client-darwin:2.3.3")
}
```

3-逻辑全部在 common

### 3.3 expect / actual 方式集成三方能力—示例:Toast 三方能力封装

1-ommonMain 定义接口

```
expect object Toast {
    fun show(message: String)
}
```

2-Android 实现

```
actual object Toast {
    fun show(message: String) {
        android.widget.Toast.makeText(
            appContext,
            message,
            android.widget.Toast.LENGTH_SHORT
        ).show()
    }
}
```

3-iOS 实现（UIKit）

```
actual object Toast {
    fun show(message: String) {
        println("iOS Toast: $message")
    }
}
```

3-说明：上层 UI 完全不关心平台

### 3.4 封装 Android / iOS 三方库的通用套路-封装步骤模板

```
1.commonMain 定义能力接口
2.androidMain 调用 Android SDK / 第三方
3.iosMain 调用 UIKit / Swift 库
4.UI 层只依赖 common
```

## 五 不能直接用的库，怎么办？

### 5.1 Android 专属库（如 Glide）

```
1-解决方案：
-Android：Glide / Coil
-iOS：SDWebImage
-common：统一 ImageLoader 接口

2-示例
expect class ImageLoader {
    fun load(url: String)
}
```

## 六 KMP 三方库集成的常见坑

### 6.1 不推荐

```
1.把 Android 库放 common
2.在 common 引用 android.*
3.业务逻辑下沉到 androidMain
4.expect 定义过于平台化
```

### 6.2 推荐

```
1.能 common 就 common
2.能接口就接口
3.UI 与平台能力解耦
```

## 七  与 Flutter / RN 的对比

|  框架   |       三方库策略       |
| :-----: | :--------------------: |
| Flutter |       Dart-only        |
|   RN    |   JS + Native Bridge   |
|   KMP   | Kotlin + expect/actual |

说明： KMP 的优势：强类型 + 原生能力

## 八 小结

```
1.KMP 能集成大量三方库
2.优先选择 Multiplatform 库
3.平台能力统一用 expect / actual
4.三方库集成决定项目上限
5.架构比库本身更重要
```
