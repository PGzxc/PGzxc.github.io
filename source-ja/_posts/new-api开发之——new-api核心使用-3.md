## 二 new-api核心


### 2.1 用户体系(基础但关键)

#### 1-New API 内置完整用户系统

```
支持：
-注册 / 登录
-第三方登录（OIDC / Telegram）
-用户分组
-用户余额
```

#### 2-用户在系统中的角色

```
用户 = AI 使用者

每个用户拥有：
-余额 
-Token(API Key)
-分组（权限）
-使用记录
```

#### 3-核心关系

```
用户 → 分组 → Token → 调用 API
```

### 2.2 Token(API Key)机制

#### 1-什么是 Token？

```
Token = 用户调用 AI 的“钥匙”
```

#### 2-调用流程

```
请求 → Token → 校验权限 → 扣费 → 返回结果
```

#### 3-创建流程

```
1.后台操作：
-创建用户
-创建 Token
-分配分组

2.示例调用（兼容 OpenAI）
curl http://localhost:3000/v1/chat/completions \
  -H "Authorization: Bearer sk-xxxx" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gpt-4o",
    "messages": [{"role": "user","content":"hello"}]
  }'
```

#### 4-Token 控制能力

```
1.可以限制：
 -调用次数
 -可用模型
 -QPS（限流）

2.用于防止：
-被刷爆 / 被盗用
```

### 2.3 钱包 & 余额机制

#### 1-钱包

```
New API 内置“钱包系统
```

#### 2-用户余额结构

```
余额 = 可消费额度

每次调用：
调用 AI → 消耗 Token → 扣余额
```

#### 3-支持方式

```
手动充值（管理员）
在线支付（第三方）
兑换码
```

#### 4-本质

```
New API = 一个“计费引擎”
```

### 2.4 计费机制(核心中的核心)

New API 支持多种计费方式：

#### 1-按 Token 消耗(最常见)

```
类似：
-OpenAI 计费方式
-根据输入 / 输出 Token 计费
```

#### 2-按请求次数

```
1次调用 = 固定费用

2.适合：
-简单 API
-图片生成
```

#### 3-模型倍率计费(重点)

1-最关键的赚钱方式:

```
用户价格 = 成本 × 倍率
```

2-示例:

|  模型  | 成本 | 倍率 | 售价 |
| :----: | :--: | :--: | :--: |
| GPT-4o |  1   |  2   |  2   |
| Claude |  1   | 1.5  | 1.5  |

3-实战建议(不同用户不同倍率):

|   用户   | 倍率 |
| :------: | :--: |
| 免费用户 |  3   |
| 付费用户 | 1.5  |

### 2.5 分组机制(最重要设计)

#### 1-分组机制

```
分组 = 权限 + 套餐 + 价格
```

#### 2-分组可以控制什么？

```
每个分组可以设置：

- 可用模型
- 调用限额
- QPS
- 价格倍率
```

#### 3-核心作用

```
一句话：
分组 = 商业化的核心抽象
```

#### 4-示例设计

| 分组 |   权限   |
| :--: | :------: |
| free | GPT-3.5  |
| pro  |  GPT-4   |
| vip  | 全部模型 |

#### 5-QPS 限制

```
防止：

- 爬虫
- 恶意调用
```

### 2.6 Token + 分组 = 权限系统

#### 1-结合来看

```
Token → 属于用户
用户 → 属于分组
分组 → 决定权限
```

#### 2-完整链路

```
请求 → Token → 用户 → 分组 → 权限校验 → 是否允许
```

### 2.7 调用完整流程

#### 1-核心流程

```
1️⃣ 用户发起请求
        ↓
2️⃣ 携带 Token
        ↓
3️⃣ 系统校验 Token
        ↓
4️⃣ 获取用户 & 分组
        ↓
5️⃣ 检查权限 / 限额
        ↓
6️⃣ 转发到模型（渠道）
        ↓
7️⃣ 获取返回结果
        ↓
8️⃣ 计算 Token 消耗
        ↓
9️⃣ 扣除余额
        ↓
🔟 记录日志
```

#### 2-本质理解

```
 New API = 请求中间层 + 计费系统
```

### 2.8 使用日志(非常重要)

#### 1-系统记录

```
- 请求日志
- Token 使用量
- 模型调用
- 用户消费
```

#### 2-用途

```
- 成本分析
- 用户行为分析
- 风控（异常调用）
```

### 2.9 安全控制(实战重点)

#### 1-Token 限制

```
- IP 限制（如支持）
- QPS 限制
```

#### 2-分组限额

```
- 免费用户限额度
- 防止刷爆
```

#### 3-日志监控

```
发现：
- 异常调用
- Key 泄露
```

## 三 常见错误

### 3.1 不做限额

```
 结果：
一夜欠几千刀 API 费
```

### 3.2 所有人用 GPT-4

```
成本爆炸
```

### 3.3 不区分用户等级

```
无法盈利
```

### 3.4 Token 泄露

```
被刷接口
```

## 四 推荐权限设计(可直接用)

### 4.1 分组方案

| 分组  |      模型      |  限额  |
| :---: | :------------: | :----: |
| free  |    GPT-3.5     | 100次  |
| basic |     GPT-4o     | 1000次 |
|  pro  | GPT-4 + Claude |  无限  |

### 4.2 收费策略

```
- 注册送额度
- 充值购买
- 月卡 / 年卡
```

## 五 总结

### 5.1 核心三大系统

```
New API 的本质：
1. 用户系统
2. 权限系统（分组）
3. 计费系统
```

### 5.2 核心一句话

```
用分组控制权限，用 Token 控制调用，用余额实现收费
```
