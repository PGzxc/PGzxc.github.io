## 二 为什么必须做风控？

### 2.1 常见事故

```
如果你已经上线 New API，但没有做风控，很容易出现：

- Token 泄露 → 被疯狂调用
- 爬虫攻击 → QPS 打满
- GPT-4 被滥用 → 一夜亏几千刀
- 恶意用户刷余额
```

### 2.2 本质问题

```
AI 接口 = 高价值资源（必须保护）
```

## 三 高级玩法

### 3.1 限流机制(第一道防线)

1-什么是限流？

```
控制请求速度（QPS）
```

2-示例

```
用户A：1秒最多 2 次请求
用户B：1秒最多 10 次请求
```

3-推荐限流策略

| 用户类型 | QPS  |
| :------: | :--: |
|   free   | 1~2  |
|  basic   | 3~5  |
|   pro    | 5~10 |
|   vip    | 10+  |

4-限流维度

```
- Token 限流
- 用户限流
- IP 限流（进阶）
```

5-作用

```
- 防止刷接口
- 防止爆账
- 防止系统被打挂
```

### 3.2 Token 安全策略

```
一、Token 是最敏感的
Token = 钱包钥匙

二、安全建议

2.1 设置调用限制
- 限制模型
- 限制额度
- 限制 QPS

2.2 定期更换 Token
类似密码：长期不换 = 高风险

2.3 分用途 Token
前端 Token ≠ 后端 Token

2.4 绑定 IP(如果支持)
防止被盗用
```

### 3.3 防刷策略

```
一、常见攻击方式
- 暴力请求 API
- 爬虫批量调用
- Token 泄露滥用

二、防刷方案

2.1限流（必须）
已讲

2.2 验证机制（前端）
- 登录验证
- 验证码
- 行为检测

2.3 请求签名（进阶）

请求 + 时间戳 + 签名
防止：重放攻击/伪造请求

2.3 黑名单机制

封禁：IP/用户/Token

2.5 异常检测
例如：1分钟 1000次请求 → 异常
```

### 3.4 日志与监控

```
1.监控什么？

- 请求量
- Token 使用量
- 用户消费
- 错误率

2.作用

- 发现异常
- 控制成本
- 优化系统

3.推荐方案

结合：
- 日志系统
- 可视化看板

企业可接入：
- Grafana
- Prometheus
```

### 3.5 防爆账策略

```
一、什么是爆账？
用户疯狂调用 → API 费用暴涨

二、防爆账方案

2.1 用户额度限制
每个用户每日上限

2.2 分组限额
free → 10万 Token
pro → 100万 Token

2.3 高价模型限制
GPT-4 → 仅 VIP 可用

2.4 余额保护
余额 = 0 → 禁止调用

2.5 实时告警（进阶）
当：消费 > 阈值 → 报警
```

### 3.6 风控系统设计(架构级)

```
1.简化版架构

请求 → 限流 → Token校验 → 权限判断 → 调用模型
           ↓
        风控检测（异常）

2.企业级架构

用户请求
   ↓
API Gateway（New API）
   ↓
风控系统（限流 + 黑名单）
   ↓
渠道（多模型）
   ↓
日志系统（监控）

3.核心模块
- 限流系统
- 黑名单系统
- 日志系统
- 告警系统
```

### 3.7 HTTPS + 安全基础

```
1.必做项
- HTTPS（SSL）
- 防止中间人攻击
- 加密 Token

2.推荐组合
- Nginx + SSL
- CDN 加速
```
