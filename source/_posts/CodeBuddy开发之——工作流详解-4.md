## 二 核心设计及三大模式

### 2.1 核心设计思想

```
CodeBuddy 把 AI 编程拆成三种“工作模式”：

不是一个工具，而是一个“开发工作流系统”
```

### 2.2 三大模式

```
Ask（问答模式）
Edit（编辑模式）
Agent（自动执行模式）
```

## 三 3种“工作模式”

### 3.1 Ask 模式(理解与分析)

```
1.作用

用于“问 AI”
- 解释代码
- 设计架构
- 排查问题
- 生成思路

2.示例:

解释这段代码的作用
这个接口为什么会报空指针异常
设计一个电商订单系统架构

3.特点

- 不直接改代码
- 只提供建议 / 分析

4.适用场景
- 学习代码
- 代码 review
- 架构设计
- Debug 思路分析
```

### 3.2 Edit 模式(代码修改核心)

```
1.作用
直接让 AI 修改代码

2.示例

在代码中写：// 将这个方法改为异步执行

AI 会：
- 修改当前函数
- 返回 diff
- 保持上下文一致

3.示例2（优化代码）
优化这段代码的性能

4. Edit 模式特点

|    能力    | 是否支持 |
| :--------: | :------: |
|  修改代码  |   支持   |
|    重构    |   支持   |
|  增强逻辑  |   支持   |
| 跨文件修改 |   有限   |

5. 适用场景

- 重构代码
- 修 Bug
- 优化逻辑
- 增加功能
```

### 3.3 Agent 模式(自动开发模式)

```
1.什么是 Agent？
AI 自动执行“多步骤开发任务”

2.示例
实现一个用户登录系统（JWT + MySQL）

3.Agent 会自动做：

-创建目录结构
-生成代码文件
-写业务逻辑
-处理依赖关系
-修复错误
-生成可运行项目

4.本质能力
从“写代码”变成：自动完成一个开发任务
```

### 3.4 三种模式对比

| 模式  |     能力     | 是否改代码 |
| :---: | :----------: | :--------: |
|  Ask  |     分析     |     否     |
| Edit  |   修改局部   |     是     |
| Agent | 自动完成项目 | 是（全局） |

## 四 三种模式如何配合使用

```
官方推荐工作流：

1. Ask（先理解）
这个系统如何设计更合理？

2. Edit（局部修改）
优化这个 Service 层

3.Agent（整体生成）
生成完整用户模块


4.正确流程（非常重要）
不是“直接让 AI 写”
而是：理解 → 修改 → 自动生成
```

## 五 真实开发场景演示

### 5.1 场景

```
做一个订单系统
```

### 5.2 步骤

```
1：Ask（设计）
设计一个订单系统结构（Spring Boot）
输出：
- 分层架构
- 表设计
- API 设计

2：Edit（优化代码）
优化 OrderService 事务处理

3：Agent（生成完整模块）
实现订单模块（创建 + 查询 + 删除）
```

## 六 使用注意事项

### 6.1 Agent 会改多个文件

```
一定要：git commit -m "before agent run"
```

### 6.2 避免一次性大需求

```
错误：做一个完整电商系统
正确：先做用户模块
```

### 6.3 Edit 模式要有上下文

```
@OrderService 优化事务逻辑
```

### 6.4 Agent 不等于生产上线代码

```
 需要：
- review
- 测试
- 安全检查
```

## 七 本篇总结

### 7.1  三层开发模型

```
Ask → 理解问题
Edit → 修改代码
Agent → 自动完成任务
```

### 7.2 本质升级

```
从：
人写代码
变成：
人定义目标 + AI 执行开发
```
