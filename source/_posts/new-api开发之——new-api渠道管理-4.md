## 二 new-api渠道

### 2.1 什么是渠道(Channel)

```
1-渠道 
= 模型来源（上游服务商）

2-常见渠道：
- OpenAI（GPT 系列）
- Anthropic（Claude）
- Google（Gemini）
- Azure OpenAI
- Midjourney Proxy
- 本地模型（Ollama）

3-本质理解：
New API = 中间层
Channel = 上游 API

4-所有请求最终都会：
转发到某个 Channel
```

### 2.2 添加渠道(实操步骤)

1-后台路径：

```
渠道管理 → 新增渠道
```

2-核心参数说明：

|   参数   |           说明           |
| :------: | :----------------------: |
|   类型   | OpenAI / Claude / Gemini |
| API Key  |         上游 Key         |
| Base URL |         API 地址         |
| 模型列表 |        支持的模型        |

3-示例(OpenAI)

```
类型：OpenAI
Base URL：https://api.openai.com/v1
Base URL:http://host.docker.internal:11434
API Key：sk-xxxx
模型：
- gpt-4o
- gpt-3.5-turbo
```

4-示例(自定义代理)

```
Base URL：https://your-proxy.com/v1

可接：
- 反代服务
- 第三方平台
- 自建网关
```

### 2.3 渠道与模型关系

1-重点理解:

```
模型 ≠ 渠道
```

2-举例:

|    渠道     |   模型   |
| :---------: | :------: |
| OpenAI账号A |  GPT-4   |
| OpenAI账号B |  GPT-4   |
| Claude账号  | Claude-3 |

3-说明

```
同一个模型可以有多个渠道！
```

4-意义：

```
 这是实现：

- 负载均衡
- 防封号
- 高可用

的基础
```

## 三 渠道管理

### 3.1 权重分流(负载均衡)

```
1.什么是权重？
控制流量分配比例

2. 示例

| 渠道 | 权重 |
| ---- | ---- |
| Key1 | 1    |
| Key2 | 2    |

请求分配：Key1：33%，Key2：66%

3.使用场景

- 主 Key + 备用 Key
- 不同成本 Key 分流
- 不同地区 Key
```

### 3.2 多 Key 轮询(核心能力)

```
1.什么是轮询？
多个 Key 轮流使用

2.工作流程
请求1 → Key1
请求2 → Key2
请求3 → Key3


3.优势
- 降低单 Key 压力
- 降低封号风险
- 提高吞吐量
```

### 3.3 自动重试(容灾机制)

```
1.工作流程

请求 → Key1（失败）
      ↓
   Key2（重试成功）

2. 常见失败原因
- Key 被封
- 限流（429）
- 网络错误

3.New API 会自动：
- 切换渠道
- 重试请求
```

### 3.4 防封号策略(实战重点)

```
一、为什么会封？
以 OpenAI 为例：
- 高频请求
- 多 IP 异常
- 非正常使用模式

二、防封策略（强烈建议）

2.1 多 Key 分散：

不要只用一个 Key！
10个 Key > 1个 Key

2.2 设置 QPS 限流

控制每秒请求数：
- 免费用户：1~2 QPS
- 付费用户：5~10 QPS

2.3 分组隔离

免费用户 → 低成本 Key
付费用户 → 高质量 Key

2.4 权重控制

核心 Key 少用：
贵 Key → 权重低
便宜 Key → 权重大

2.5 日志监控

及时发现：
- 异常请求
- Token 泄露
```

### 3.5 成本优化策略

1-不同模型成本差异

|   模型   | 成本 |
| :------: | :--: |
|  GPT-4   |  高  |
| GPT-3.5  |  低  |
| 本地模型 | 极低 |

2-优化策略

```
1.分层模型：
简单问题 → GPT-3.5
复杂问题 → GPT-4

2.本地模型兜底

使用：
- Ollama
- 本地 LLM
优势：
- 0 成本
- 无限调用

3.动态路由（进阶）

根据(自动选择模型)：
- 用户等级
- 请求复杂度
```

## 五 常见坑

### 5.1 只配置一个 Key

```
 结果：一封全挂
```

### 5.2 不做重试

```
结果：请求失败率高
```

### 5.3 不限流

```
结果：被封号 / 爆账
```

```
结果：成本失控
```
