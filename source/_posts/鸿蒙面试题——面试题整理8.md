## 二 面试题解答(仅供参考)

### 2.1 鸿蒙应用打包后的文件扩展名是什么？

```
打包后的文件扩展名为.hap(HarmonyOS Ability Package) ，这是 HarmonyOS 应用的标准包格式。
一个.hap 文件包含了应用的代码、资源、配置文件等所有运行所需的内容，
方便在鸿蒙设备上进行安装和运行。
```

### 2.2 ArkTS 是什么，它与 TypeScript 有什么区别？

```
1、ArkTS是什么？
ArkTS 是 HarmonyOS 优选的主力应用开发语言 ，
它保持了 TypeScript 的基本风格，同时通过规范定义强化开发期静态检查和分析，
提升程序执行稳定性和性能。


2、与 TypeScript 相比，ArkTS 的区别主要体现在：

2-1 扩展了 UI 相关能力：
定义了声明式 UI 描述、自定义组件、事件方法、属性方法；
提供了多维度的状态管理机制；
支持控制渲染、循环渲染的能力 。

2-2 强化了检查机制：
不支持 var、any、unknown、Symbol；
不支持解构赋值；
不支持使用对象字面量进行类型声明；
不支持在运行时动态增删对象的属性；
不支持在函数内声明函数；
不支持使用 typeof 作为类型；
不支持使用 #符号开头声明的私有字段，改用 private 关键字；
不支持把 function 定义函数赋值给变量，改为使用箭头函数 。
```

### 2.3 请简述页面和自定义组件的生命周期有哪些？

```
1、页面生命周期（被 @Entry 装饰的组件生命周期）：

onPageShow：页面每次显示时触发一次，包括路由过程、应用进入前台等场景。
onPageHide：页面每次隐藏时触发一次，包括路由过程、应用进入后台等场景。
onBackPress：当用户点击返回按钮时触发。

2、组件生命周期（用 @Component 装饰的自定义组件的生命周期）

aboutToAppear：组件即将出现时回调，在创建自定义组件的新实例后，执行其 build () 函数之前执行。
onDidBuild：组件 build () 函数执行完成之后回调，可用于埋点数据上报等不影响实际 UI 的功能，
但不建议在此函数中更改状态变量、使用 animateTo 等功能 。

aboutToDisappear：在自定义组件析构销毁之前执行，不允许在此函数中改变状态变量 
```

### 2.4 如何在鸿蒙应用中进行数据持久化？

```
1、用户首选项（Preferences）：
轻量级配置数据持久化方式，适合保存应用配置信息、用户偏好设置等。
以文本形式保存数据，数据全量加载到内存，访问速度快，但不适用于存储大量数据 。

2、键值型数据库（KV - Store）：
用于存储结构简单的数据，如商品名称和价格、员工工号和出勤状态等。
以 “键值对” 形式组织数据，适用于数据关系不复杂的场景 。

3、关系型数据库（RelationalStore）：
基于 SQLite，适用于存储包含复杂关系的数据，如学生信息、雇员信息等。
提供一系列 SQL 操作，方便进行数据的增删改查 。
```

### 2.5 父子组件如何通信？

```
1、@Prop 装饰器实现父子单向同步：
@Prop 装饰的变量可和父组件建立单向同步关系，该变量可变，但变化不会同步回父组件 。

2、@Link 装饰器实现父子双向同步：
子组件中被 @Link 装饰的变量与其父组件中对应的数据源建立双向数据绑定 。

3、@Provide 和 @Consume 装饰器实现与后代组件双向同步：
应用于状态数据在多个层级之间传递的场景，摆脱参数传递机制束缚，实现跨层级传递。
@Provide 装饰的状态变量对其所有后代组件可用，后代通过 @Consume 获取该变量，建立双向数据同步 。

4、@Observed 和 @ObjectLink 装饰器用于嵌套类对象属性变化场景：
对于多层嵌套的情况，如二维数组、数组项 class、class 的属性是 class 等，
当第二层属性变化无法被常规装饰器观察到时，可使用这两个装饰器。
被 @Observed 装饰的类，其属性变化可被观察到；
子组件中 @ObjectLink 装饰器装饰的状态变量用于接收 @Observed 装饰的类的实例，
和父组件中对应的状态变量建立双向数据绑定 。
```

### 2.6 兄弟组件如何通信？

```
1、通过公共父组件传递：
若两个组件是同一个父组件的子组件，父组件可作为中介传递数据或事件。
例如，父组件在其内部定义数据和方法，一个子组件调用父组件方法修改数据，
另一个子组件监听父组件数据变化做出响应 。

2、使用全局状态管理：
借助全局状态管理工具，如 AppStorage、LocalStorage 来存储共享数据。
兄弟组件可独立读取和更新这个全局状态，从而实现通信。
例如，在一个兄弟组件中向 AppStorage 存入数据，另一个兄弟组件从 AppStorage 中获取该数据并处理 。
```

### 2.7 如何实现页面间的通信？

```
1、使用 @Provide 和 @Consume 装饰器：
在一个页面的顶层组件用 @Provide 提供状态，
另一个页面的组件通过 @Consume 获取该状态，实现跨页面的数据共享和通信 。

2、路由跳转传参：
在进行路由跳转时，通过在 URL 中携带参数的方式传递数据到目标页面。
例如，使用 Router.pushUrl ('/targetPage?param1=value1&param2=value2') 进行跳转，
目标页面可解析 URL 获取参数 。

3、导航跳转传参：
利用鸿蒙提供的导航组件进行跳转时传递参数。
例如，在使用 Navigation.push ({uri: '/targetPage', params: {data: ' 要传递的数据 '}}) 跳转时，
目标页面可在 onPageShow 等生命周期中获取传递的参数 。
```

### 2.8 跨设备通信的方式有哪些？

```
1、分布式软总线：
一种高性能通信机制，允许设备之间建立直接连接，进行低延迟、高效率的数据传输，实现设备间的协同工作 。

2、蓝牙：使用标准的蓝牙技术进行设备间短距离通信，适用于一些对传输速率要求不高、距离较近的场景 。
3、WLAN：通过 WLAN 网络实现设备间通信，传输速率相对较高，适用于需要传输大量数据或对实时性要求较高的场景 。
4、远程服务调用：通过分布式任务调度实现跨设备的服务调用，一个设备上的应用可以调用其他设备上提供的服务 。
```

### 2.9 如何进行全局状态管理？

```
1、@Provide 和 @Consume 装饰器：
适用于整个组件树而言 “全局” 的状态共享，且该状态改动不频繁的场景。
在最顶层组件中使用 @Provide 装饰器提供状态，其他需要共享状态的组件通过 @Consume 装饰器获取该状态 。
优点是减少了状态传递的层级，提升了代码的可维护性和可拓展性。
但要注意确保状态的生命周期与组件的生命周期相匹配 。

2、使用全局存储对象：
例如 AppStorage，它可用于存储应用级别的全局状态数据。
在应用的任何地方都能访问和修改 AppStorage 中的数据，从而实现全局状态管理。
在使用时需注意数据的一致性和数据更新时的通知机制，确保相关组件能及时响应数据变化 。
```

### 2.10 鸿蒙系统的微内核、分布式软总线、分布式数据管理和分布式任务调度分别有什么作用，在开发项目中如何利用这些特性？

```
1、微内核：
1-1 作用：
提供最基础的操作系统服务，如进程管理、线程管理、内存管理、中断管理等。采
用微内核设计，将系统服务尽量细化并独立，服务之间通过消息通信，增强了系统的安全性、稳定性和可扩展性 。
不同的服务运行在独立的进程空间，一个服务出现故障不会影响其他服务，便于系统的维护和升级 。

1-2 开发利用：
开发者在开发应用时，无需过多关注底层内核的复杂实现细节，系统提供的稳定内核服务保障了应用运行的基础环境。
例如，应用中的多线程开发，依赖内核的线程管理机制来创建、调度和销毁线程，开发者只需调用相应的 API 即可 。

2、分布式软总线：
2-1 作用：
实现设备之间的自动发现、连接和通信，屏蔽了不同设备的硬件差异和网络细节，
为上层应用提供统一的分布式通信能力，让不同设备如同在一个虚拟总线上进行数据交互 。
可用于构建分布式应用，实现设备间的协同，如多设备文件共享、分布式游戏等 。

2-2 开发利用：
开发者在开发分布式应用时，通过调用分布式软总线的 API，可方便地实现设备间的数据传输和交互。
例如，开发一个跨设备的文件传输应用，利用分布式软总线可快速发现周围可连接的设备，并建立连接进行文件数据传输 。

3、分布式数据管理：
3-1 作用：
对跨设备的数据进行统一管理和访问，支持数据的同步、共享和协同操作。
能够保证数据在不同设备上的一致性，提供高效的数据存储和查询功能 。
比如在多个设备上使用的笔记应用，通过分布式数据管理可确保在任意设备上对笔记的修改能及时同步到其他设备 。

3-2 开发利用：
在开发涉及多设备数据共享的应用时，借助分布式数据管理 API，可方便地对共享数据进行增删改查操作，
并且无需担心数据同步和一致性问题。
例如，开发一个团队协作的项目管理应用，团队成员在各自设备上对项目任务数据的更新能实时同步到其他成员设备 。

4、分布式任务调度：
4-1 作用：
根据设备的资源情况（如 CPU、内存、存储等）和任务需求，合理地将任务分配到最合适的设备上执行，
实现跨设备的任务协同处理，提高系统整体的资源利用率和任务执行效率 。
例如，对于一个复杂的图像渲染任务，可将计算量较大的部分分配到具有高性能 GPU 的设备上执行 。

4-2 开发利用：
开发者在开发需要跨设备协同处理任务的应用时，通过分布式任务调度 API，
可指定任务的属性和需求，系统会自动将任务调度到合适设备。
如开发一个分布式计算应用，可将不同的计算子任务分发到不同设备并行计算，最后汇总结果 。
```


### 2.12 在鸿蒙应用开发中，如何进行性能优化？

```
1、优化应用启动速度：
减少启动时不必要的初始化操作，将一些非关键的初始化任务延迟到应用启动完成后异步执行；
对启动过程中的资源加载进行优化，如图片资源采用合适的格式和大小，避免加载过大资源；
合理使用懒加载机制，对于首次启动不需要立即展示的界面或数据，推迟加载 。

2、内存优化：
及时释放不再使用的对象和资源，避免内存泄漏。
例如，在 Activity 或 Ability 销毁时，释放相关资源；
优化数据结构的使用，避免创建过多不必要的对象；
使用对象池技术，对于频繁创建和销毁的对象，复用对象而不是每次都重新创建 。

3、布局优化：
简化布局层级，避免过度嵌套布局，减少 UI 渲染的计算量；
使用合适的布局容器，根据界面需求选择 LinearLayout、RelativeLayout 等，以提高布局效率 。

4、资源管理：
对图片、音频、视频等资源进行合理压缩和优化，选择合适的资源加载方式，如使用缓存机制减少重复加载 。
例如，对于频繁使用的图片，将其缓存在内存或本地存储中，下次使用时直接读取缓存 。

5、多线程与异步处理：
对于耗时操作，如网络请求、数据计算等，使用多线程或异步任务处理，避免阻塞主线程，保证 UI 的流畅性 。
例如，在进行网络请求获取数据时，将请求操作放在子线程中执行，主线程继续处理其他 UI 相关事务 。
```
