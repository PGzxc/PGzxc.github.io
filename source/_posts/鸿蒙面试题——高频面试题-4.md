## 三 高频面试题解答(仅供参考)

### 3.1 分布式能力开发：分布式数据，分布式软总线

1、请解释一下 HarmonyOS 的分布式软总线和分布式数据能力

```
1、分布式软总线(Distributed Soft Bus)
-它是HarmonyOS分布式能力的基础，可以理解为一张无形的“网”，将不同设备的硬件能力虚拟化、池化。
-当设备处于同一网络环境下，它能让设备之间像连接在一条总线上一样，进行高效、安全的数据传输和任务调用。
-开发者无需关心底层复杂的网络连接方式(如 Wi-Fi、蓝牙)，可以直接使用分布式软总线提供的 API 进行跨设备通信。

2、分布式数据(Distributed Data Service)
-鸿蒙提供了分布式数据管理服务，它允许应用在不同设备间共享和同步数据。
-开发者可以将数据存储在一个逻辑上“统一”的数据库中，即使这些数据物理上分散在不同的设备上。
-当一个设备上的数据发生变化时，其他设备上的数据会通过分布式数据服务自动同步。
-这使得跨设备协作的应用开发变得非常简单，例如，你可以在手机上编辑文档，然后在平板上无缝继续。
```

2、分布式通信时，鸿蒙如何保障数据安全？

```
鸿蒙在分布式通信中采用了多重安全机制：
-端到端加密： 设备之间的数据传输默认采用加密，保障数据在传输过程中的机密性。
-设备认证： 只有经过认证的、可信的设备才能加入分布式网络，防止未经授权的设备接入。
-访问控制： 应用在访问其他设备的数据或能力时，需要经过用户授权和权限校验，确保数据只能被合法应用访问。
-安全沙箱： 每个应用都运行在独立的沙箱中，即使某个应用被攻破，也难以影响到系统或其他应用的数据。
```

3、分布式设备开发指南

```
1.分布式数据同步
2.设备发现开发
3.设备绑定开发
4.设备信息查询开发
5.设备上下线监听开发
```

### 3.2 arkts/ts，arkui，ability框架及分布式api

1、ArkTS与TypeScript的主要区别是什么？

```
ArkTS在TS基础上添加HarmonyOS特定语法，如装饰器（@Component、@State）和并发支持（@Concurrent）。
TS更通用，ArkTS优化为多端适配。
示例：@Component struct MyComp { @State count: number = 0; }。
实践：使用@Entry定义入口
```

2、ArkUI框架的核心原理是什么？如何与Ability集成？

```
原理：基于组件树和数据绑定，渲染引擎高效更新DOM。
核心组件：Text、Button等，支持动画/布局。
集成：Ability的onWindowStageCreate加载ArkUI根节点，如windowStage.loadContent('pages/Index')。
实践：@Builder自定义UI块，提高复用。
```

3、分布式API如何在Ability中实现跨设备调用？

```
使用ohos.ability.distributedAbilityManager绑定远程Ability。
原理：通过Want指定deviceId，实现RPC调用。
示例：startAbility(want: { deviceId: 'remoteId', bundleName: 'com.example.remote' })。
实践：处理连接失败回调，确保安全性。
```

4、总结

```
ArkTS扩展TypeScript，支持装饰器和响应式编程；
ArkUI是声明式UI框架；
Ability是应用组件抽象
```


### 3.4 组件化设计模式

1、Ability 组件化

```
鸿蒙的 Ability 框架本身就是一种组件化的体现。
一个应用可以由多个 Ability 组成，每个 Ability 负责不同的功能，
例如一个处理用户登录，另一个负责展示商品列表
```

2、ArkUI 组件化

```
ArkUI 的 @Component 装饰器是组件化最直接的实践。
开发者可以将复杂的 UI 界面拆分成多个小的、可复用的自定义组件。
例如，一个商品列表项可以封装成一个自定义组件，然后在多个页面中复用。
这遵循了“高内聚，低耦合”的设计原则。
```

3、模块化(HAP/Shared Library)

```
在更宏观的层面，鸿蒙应用可以被拆分成多个 HAP（HarmonyOS Ability Package）包或者共享库。
例如，可以将一个应用的登录、支付等功能独立成一个模块，供其他应用或自身应用的其他部分调用
```

### 3.5 原子化服务、卡片服务

1、基本概念

```
1、原子化服务的特点？
-无需安装，点开即用，体积小。
-支持 卡片形态，嵌入桌面或服务中心。
-与分布式能力结合，可跨设备拉起。

2、卡片服务实现？
-通过 FormAbility 提供 UI 卡片，开发者定义卡片布局与交互。
-生命周期：Create → Update → Delete。
```

2、开发相关

```
1、原子化服务与传统服务的区别是什么？如何开发？
区别：免安装、轻量；传统需完整HAP。
开发：manifest中type: 'atomicService'，@FormComponent定义卡片。
示例：postCardAction({ action: 'refresh' })。
实践：天气服务卡片。

2、服务卡片如何实现交互和更新？
通过事件监听和updateForm API。
示例：on('message', (msg) => { updateForm(options); })。
优化：集成分布式数据实时同步。

3、原子化服务在桌面场景的应用案例？
桌面卡片提供快捷入口。
示例：配置formConfig支持多尺寸，拖拽交互。
```

3、它们的关系

```
原子化服务（Meta Service）支持免安装，
卡片（Form）是其UI形式
```

### 3.6 鸿蒙系统特性：ability,fa/stage模型，多线程，进程

1、FA模型与Stage模型的区别是什么？

```
FA：组件独立进程，高隔离；
Stage：共享进程，高效共享。
示例：Stage中Context全局访问。
选择：复杂App用Stage。
```

2、鸿蒙多线程方式？

```
TaskPool：轻量线程池
Worker：与 JS/ArkTS 隔离的线程
适用场景：TaskPool 适合并行计算，Worker 适合长生命周期后台任务。
```

3、鸿蒙进程与线程模型如何支持分布式？

```
进程资源隔离，线程共享执行。
分布式：RPC跨进程。
示例：@Concurrent装饰方法支持并发。
```

4、一句话总结

```
FA(独立VM)和Stage(共享VM)模型
```

### 3.7 openharmony应用模型和生命周期

1、OpenHarmony应用模型的核心组件是什么？

```
Ability和Context，支持FA/Stage
```

2、UIAbility的生命周期顺序是什么？

```
1、生命周期
onCreate → onWindowStageCreate → onForeground → 
onBackground → onWindowStageDestroy → onDestroy。

2、实践：
onDestroy释放资源。
```

3、自定义组件生命周期与页面级的区别？

```
1、组件：
aboutToAppear/aboutToDisappear；

2、页面：onPageShow/onPageHide。

3、示例：组件用于局部动画。
```

### 3.8 mvvm架构模式

1、鸿蒙中如何体现 MVVM？

```
View：ArkUI 声明式 UI。
ViewModel：状态管理（@State/@Link/@Observed）。
Model：业务数据与分布式数据存储。
```

2、MVVM与MVC的区别及优势？

```
MVVM自动更新，MVC手动。
优势：测试友好，减少代码。
```
